# Course Concepts Map

How AI Product Factory demonstrates the course concepts, and where to find each one in the repository or running demo.

| Course concept | Where it is demonstrated | Status |
|---|---|---|
| Multi-agent system | Typed `IAgent` modules (`apps/web/src/server/agents/`) chained by a server-side orchestrator behind `POST /api/blueprint`. The orchestrator now branches on `generationMode`: the deterministic agent chain for `"demo"`, or a single Gemini call (`llm/blueprint-generator.ts`) for `"live"` — see the LLM/Gen AI row below | Orchestrator wired (Phase 3, demo-mode output); real LLM calls now implemented behind an opt-in Live Gemini Mode |
| LLM / Generative AI integration | "Live Gemini Mode": an opt-in, server-gated switch (`GenerationModeSwitcher` in the UI) lets the user choose deterministic Demo Mode (default) or a real Gemini call for Architecture/Security/Roadmap/Tasks, using the user's final selected-skill guidance as prompt context (`skill-context.ts` + `blueprint-prompt.ts`). Readiness Score is always computed by the deterministic Evaluation agent in both modes, never by Gemini. Gated by `ENABLE_LIVE_AI`/`LLM_PROVIDER`/`LLM_API_KEY`, capped at `LIVE_AI_DAILY_LIMIT` (default 10) requests/day per anonymous cookie identity, behind an `ILlmProvider` abstraction (`llm/types.ts`) so other providers could be added later. A `GenerationLoadingBanner` shows mode-specific loading copy (Live: "may take up to 30 seconds," lists skills used as context) during generation | New: implemented, server-side enforced, Demo Mode remains default and always available; verified end-to-end including failure paths (unavailable/rate-limited/timeout/invalid output) and consistent Readiness Score structure across both modes |
| Agent skills | `agent-skill-kit/skills/` used by this project per `AGENTS.project.md` skill routing. Recommended skills are surfaced to the user via an editable `SkillSelector` (keep/add/remove/reset, `spec-driven-development` protected); the **final, user-adjusted** selection — not just the recommendation — deterministically changes the generated Architecture/Security/Roadmap/Tasks/Readiness Score via `skill-enrichment.ts`'s skill-id-keyed rules (8 skills covered: `agent-security-review`, `database-design-rules`, `testing-patterns`, `observability-rules`, `mcp-tool-consumption`, `agentic-commerce-rules`, `a2a-agent-design`, `a2ui-patterns`) | In place; manual selection + deterministic enrichment verified end-to-end, no LLM call |
| MCP Server / Tool Integration | Public MCP server (`packages/mcp-skill-server`, Streamable HTTP transport via `@modelcontextprotocol/sdk`) exposing `list_skills`, `get_skill`, `recommend_skills`, `score_readiness` over the shared `packages/skill-tools` read-only layer. Deployed to Render at `https://ai-product-factory-mcp.onrender.com`. `apps/web` now consumes `recommend_skills` over the network MCP-first (`MCP_SERVER_URL`), with an automatic local fallback (`src/server/mcp-client.ts` + `src/server/orchestrator.ts`) — the first real client-to-deployed-server MCP call in this project, not just an in-process call into the same logic | Phase 4A, 4B, and the Phase 6A remote-MCP-first follow-up all done; verified end-to-end with a real MCP client and with the live Render deployment |
| Security | Security agent (`apps/web/src/server/agents/security.ts`) detects signals (data sensitivity flags + keyword matches via `security-signals.ts`) and builds tailored Data Classification, full 6-category STRIDE coverage, Security Risks, Approval-Required Actions, and Recommendations sections — content varies per idea, not a fixed template | Phase 5 done (deterministic, rule-based; no LLM call) |
| Human-in-the-loop | Product Spec approval step blocks Architecture/Security/Roadmap generation until the user approves; enforced by the orchestrator's two-stage API contract (`spec` vs. `blueprint`). A Live Gemini failure surfaces a clear message with a one-click "Switch to Demo Mode" action rather than silently failing or substituting unannounced content | Implemented in UI (Phase 2) and orchestrator (Phase 3); failure-recovery path added with Live Gemini Mode |
| Specification-driven development | Product Spec → Architecture → Roadmap flow; `spec-driven-development` skill used for planning this repo's features, and is the one skill that's always selected for every generated blueprint (`PROTECTED_SKILL_IDS`) | In place |
| Deployability | Vercel-first deployment plan, optional Render Free MCP server, public GitHub repo; monorepo is deployment-ready (root `README.md`, `next.config.mjs` file-tracing fix, verified build/start commands from repo root) | Phase 6A done — deployment-ready and verified locally; not yet actually deployed (no live URLs yet) |
| Evaluation | Evaluation agent (`apps/web/src/server/agents/evaluation.ts`) reuses `packages/skill-tools`' `scoreReadiness` (the same heuristic the public MCP server's `score_readiness` tool exposes) to compute real component scores and a final score, plus a derived interpretation naming the weakest component, a "Skills Applied" list, and a "How to Improve This Score" section (`readiness-recommendations.ts`) with actionable, signal-driven skill and process-safeguard recommendations that update when the user changes their skill selection. **Runs unconditionally in both Demo and Live Gemini modes** — the orchestrator always calls this agent on whichever Architecture/Security/Roadmap/Tasks text was just produced, so Readiness Score is consistently rich either way; Gemini is never asked to generate it | Phase 5 + this pass done (deterministic heuristic and recommendations, shared with MCP server; not a live evaluation model) |

## Notes

- "Demo mode" outputs are deterministic and committed as the canonical example in `examples/generated-blueprint/` — these are not regenerated by Live Gemini Mode, which produces different (non-deterministic) output by design.
- Phases are tracked in `AI_Product_Factory_PROJECT_PLAN.md` §15 and `AGENTS.project.md`.
- See `docs/architecture.md`, "Live Gemini Mode," for the full design, the daily-quota mechanism and its documented in-memory/serverless limitations, and the verification performed.
